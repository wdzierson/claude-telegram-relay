# Agent Architecture

> Detailed design for each module. Each abstraction should be describable in 2-3 sentences.
> Updated as modules are implemented.

## Module Map

### `src/config/` — Configuration

Loads `.env` and `config/profile.md` at startup. Exports a single typed `Config` object consumed by all other modules. Validates required values and provides sensible defaults.

```typescript
interface Config {
  telegram: {
    botToken: string;
    allowedUserId: string;
  };
  claude: {
    path: string;          // CLI binary path
    projectDir?: string;   // Working directory for Claude
  };
  supabase?: {
    url: string;
    anonKey: string;
  };
  voice?: {
    provider: 'groq' | 'local';
    groqApiKey?: string;
    whisperBinary?: string;
    whisperModelPath?: string;
  };
  user: {
    name: string;
    timezone: string;
  };
  paths: {
    relayDir: string;      // ~/.bright
    tempDir: string;
    uploadsDir: string;
  };
}
```

### `src/agent/` — Orchestrator

The core loop. Receives a user message (text, transcribed voice, image caption), assembles context (profile, memory, time), calls Claude, parses memory intents from the response, and returns the clean reply. This is the "brain" — everything else is I/O.

```
UserMessage → [assemble context] → [call Claude] → [parse intents] → CleanResponse
```

**Key functions:**
- `handleMessage(message: IncomingMessage): Promise<string>` — main entry point.
- `buildPrompt(text, relevantContext, memoryContext): string` — assembles the full prompt.
- `callClaude(prompt, options): Promise<string>` — spawns CLI, returns output.

**Types:**
```typescript
interface IncomingMessage {
  type: 'text' | 'voice' | 'photo' | 'document';
  text: string;              // Original text, or transcription, or caption
  filePath?: string;         // For images/documents
  metadata?: Record<string, unknown>;
}
```

### `src/channels/telegram/` — Telegram Channel

grammY bot that listens for messages, converts them to `IncomingMessage`, passes to the agent, and sends the response back. Handles Telegram-specific concerns: file downloads, typing indicators, message length splitting, authorization.

**Responsibilities:**
- Bot setup and middleware (auth check).
- Message handlers: text, voice, photo, document.
- File download (voice OGG, images, documents to temp dir).
- Response formatting and chunking (4096 char limit).
- Lock file management (single instance).

Does NOT contain business logic or prompt assembly.

### `src/memory/` — Memory System

Manages persistent storage in Supabase. Three capabilities:
1. **Save messages** — insert into `messages` table (embeddings auto-generated by webhook).
2. **Get context** — retrieve facts and active goals for prompt injection.
3. **Semantic search** — call the `search` Edge Function to find relevant past messages.
4. **Process intents** — parse `[REMEMBER]`, `[GOAL]`, `[DONE]` tags from Claude's response.

**Public API:**
```typescript
// Save a message to history
saveMessage(role: string, content: string, metadata?: Record<string, unknown>): Promise<void>

// Get facts + goals for prompt context
getMemoryContext(): Promise<string>

// Semantic search for relevant past messages
getRelevantContext(query: string): Promise<string>

// Parse and save memory intents, return cleaned response
processMemoryIntents(response: string): Promise<string>
```

All functions are no-ops if Supabase is not configured (graceful degradation).

### `src/voice/` — Voice Transcription

Routes audio buffers to the configured provider (Groq or local whisper.cpp). Returns transcribed text. Stateless and provider-agnostic from the caller's perspective.

```typescript
transcribe(audioBuffer: Buffer): Promise<string>
```

Future: TTS (text-to-speech) will be added here as a second capability.

### `src/scheduler/` — Scheduled Tasks

Standalone scripts for proactive behavior. Each script:
1. Gathers context (goals, calendar, last activity).
2. Asks Claude whether to act.
3. Sends a Telegram message if Claude says yes.

Currently two patterns:
- **Smart check-in** — runs every 30 min, Claude decides if/what to say.
- **Morning briefing** — runs once daily, assembles a summary from data sources.

These run as separate processes (launched by cron/launchd/PM2), not inside the main relay.

### `src/utils/` — Utilities

Shared helper modules used across channels and agent layers.

- `src/utils/extract.ts` — Text extraction for PDFs (pdf-parse), Word docs (mammoth), plain text, and audio transcription routing. Returns null for images (handled by vision).

## Data Flow

```
User sends Telegram message
  │
  ▼
telegram/bot.ts
  ├── Auth check (middleware)
  ├── Download files if needed
  ├── Transcribe voice if needed (→ voice/)
  ├── Build IncomingMessage
  │
  ▼
agent/index.ts — handleMessage()
  ├── memory.getRelevantContext(text)     ─→ Supabase Edge Function
  ├── memory.getMemoryContext()           ─→ Supabase RPC
  ├── agent/prompt.ts — buildPrompt()
  ├── agent/claude-cli.ts — callClaude()  ─→ spawns `claude -p ...`
  ├── memory.processMemoryIntents()       ─→ parse tags, save to Supabase
  │
  ▼
telegram/send.ts
  ├── Split long messages
  ├── Send via ctx.reply()
  │
  ▼
memory.saveMessage() — persist both user & assistant messages
```

## Session Management

Claude CLI supports `--resume <sessionId>` for conversation continuity. Session ID is stored in `~/.bright/session.json`. Each message resumes the prior session so Claude has conversational context beyond what we inject.

## Database Schema (Supabase)

Four tables:
- **messages** — conversation log with optional `embedding VECTOR(1536)`.
- **memory** — facts, goals, preferences with embeddings.
- **logs** — observability (level, event, message, duration).
- **attachments** — uploaded files (images, documents, audio) with storage URLs, extracted text, and embeddings. Searchable via `match_attachments` RPC and `search_attachments` tool.

Two Edge Functions (Deno, deployed to Supabase):
- **embed** — triggered by INSERT webhook on messages/memory, generates OpenAI embedding.
- **search** — accepts a query, generates embedding, runs `match_messages` or `match_memory` RPC.

Four RPC functions: `get_recent_messages`, `get_active_goals`, `get_facts`, `match_messages`, `match_memory`.
